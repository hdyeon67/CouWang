# 쿠왕(Couwang) 프로덕트 데이터 FAQ

> 데이터 분석 진행 시 궁금하실 수 있는 부분을 미리 정리했습니다.

---

## 데이터 일반

### Q. 이 데이터는 실제 서비스 데이터인가요?
시뮬레이션으로 생성한 합성(synthetic) 데이터입니다. 실제 모바일 앱 서비스의 패턴을 참고하여 설계했습니다.

### Q. 데이터 기간과 규모는 어떻게 되나요?
2026-02-01 ~ 2026-05-01 (90일)이며, 기존 유저 500명에 매일 신규 유저가 유입되는 구조입니다. CSV 9개 파일, 총 약 34MB입니다.

### Q. 각 테이블의 크기는 어떻게 되나요?

| 파일 | 행 수 | 설명 |
|------|-------|------|
| `users.csv` | ~4,900 | 유저 정보 |
| `coupons.csv` | ~13,400 | 쿠폰 정보 |
| `memberships.csv` | ~6,100 | 멤버십 카드 |
| `event_logs.csv` | ~235,000 | 유저 행동 로그 (43종 이벤트) |
| `notifications.csv` | ~2,100 | 만료 알림 발송/열람 |
| `sessions.csv` | ~54,600 | 앱 세션 |
| `ab_assignments.csv` | ~4,800 | A/B 실험 배정 |
| `brands.csv` | 14 | 브랜드 메타데이터 |
| `campaigns.csv` | 5 | 마케팅 캠페인 |

### Q. 테이블이 9개인데 어디서부터 봐야 하나요?
분석 목적에 따라 다릅니다:
- 퍼널 분석 → `event_logs` + `users`
- 쿠폰 분석 → `coupons` + `brands`
- 알림 효과 → `notifications` + `event_logs`
- 멤버십 분석 → `memberships` + `users`
- A/B 테스트 → `ab_assignments` + `event_logs`
- 리텐션 분석 → `users` + `sessions`

### Q. 테이블 간 관계가 어떻게 되나요?

```
users ─┬─< coupons         (유저가 여러 쿠폰을 등록)
       ├─< memberships     (유저가 여러 멤버십을 등록)
       ├─< event_logs      (유저의 모든 행동 로그)
       ├─< notifications   (유저에게 발송된 알림)
       ├─< sessions        (유저의 앱 세션)
       └─< ab_assignments  (유저의 실험 배정)

coupons ─┬─< event_logs    (쿠폰 관련 이벤트)
         └─< notifications (쿠폰 만료 알림)

sessions ──< event_logs    (세션 내 이벤트)
brands ────< coupons       (브랜드별 쿠폰)
```

모든 테이블은 `user_id`로 연결됩니다. 시각적 ERD는 `schema.dbml` 파일을 [dbdiagram.io](https://dbdiagram.io)에 붙여넣으면 확인할 수 있습니다.

## 필드별 궁금한 점

### Q. `coupons.register_type`의 `manual`과 `auto`는 뭔가요?
쿠폰 등록 방식입니다. `auto`는 카메라로 스캔하여 자동 인식, `manual`은 유저가 직접 입력한 것입니다. `event_logs`에서 `coupon_auto_scan`, `coupon_manual_input` 등의 이벤트로 등록 과정을 추적할 수 있습니다.

### Q. `users.is_activated`와 `users.is_register_activated`는 각각 뭔가요?
둘 다 유저의 Aha Moment 달성 여부를 나타내는 boolean 필드입니다. 하나는 쿠폰 등록 기준, 다른 하나는 소진 기준이며, 정확한 조건(기간 등)은 데이터에서 확인하실 수 있습니다.

### Q. `users.status`가 `churned`인 유저는 어떤 기준인가요?
일정 기간 이상 앱 활동이 없으면 churn 상태로 전환됩니다. `churned_at` 필드에 이탈 시점이 기록되어 있습니다.

### Q. `notifications.notification_type`의 d30, d7, d3, d1, d0은 뭔가요?
쿠폰 만료일 기준 D-day 알림입니다. `d7`이면 만료 7일 전에 발송된 알림이고, `d0`이면 만료 당일입니다. 유저별로 어떤 타이밍의 알림을 켜고 끄는지도 설정할 수 있습니다.

### Q. `notifications.led_to_redeem`은 어떤 기준인가요?
해당 알림 이후 일정 시간 내에 쿠폰이 소진되었으면 `True`입니다.

### Q. `push_received`, `push_open`, `push_click`의 차이가 뭔가요?

세 이벤트는 푸시 알림의 단계별 유저 행동을 나타냅니다.

**`push_received`** — 알림이 유저의 기기(잠금화면/알림센터)에 **도착**한 시점입니다. 유저가 봤는지 안 봤는지는 알 수 없고, 단지 "서버에서 보냈고 기기에 전달됨"을 의미합니다.

**`push_open`** — 유저가 알림을 **탭하여 앱에 진입**한 시점입니다. 알림센터에서 해당 알림을 눌러 쿠왕 앱이 열린 것입니다. 알림을 보고도 무시하거나 스와이프로 지운 유저는 여기에 포함되지 않습니다.

**`push_click`** — 앱에 진입한 뒤, 알림이 안내한 **쿠폰을 실제로 터치하여 상세 화면을 조회**한 시점입니다. 앱은 열었지만 다른 행동을 하거나 바로 나간 유저는 여기에 포함되지 않습니다.

따라서 퍼널 순서는 `received → open → click` 이며, 각 단계에서 유저가 빠지기 때문에 숫자가 줄어드는 역삼각형 구조입니다.

### Q. `event_logs`에서 `dialog_` 이벤트는 뭔가요?
앱 내에서 "등록하시겠습니까?", "삭제하시겠습니까?" 같은 확인 팝업이 뜰 때 발생하는 이벤트입니다. `_confirm`은 확인 버튼, `_cancel`은 취소 버튼을 누른 것입니다.

### Q. `event_logs`에서 `report_banner_view`는 뭔가요?
홈 화면에 표시되는 리포트 배너(이번 달 절약 금액, 소진 현황 요약)를 유저가 본 이벤트입니다. `report_banner_click`은 배너를 클릭하여 상세 리포트로 진입한 것입니다.

### Q. `event_logs.properties` 컬럼에 JSON이 들어있는데 뭔가요?
이벤트의 부가 정보가 JSON 형태로 저장되어 있습니다. 알림 타입, 브랜드명, 필터 조건 등의 맥락 정보가 담겨 있으며, `json.loads()`로 파싱하여 활용할 수 있습니다.

### Q. `event_logs.event_name`에는 어떤 이벤트가 있나요?

| 카테고리 | 이벤트명 |
|----------|---------|
| 앱 라이프사이클 | `app_install`, `app_signup_start`, `app_signup_complete`, `app_onboarding_complete` |
| 쿠폰 등록 | `coupon_register_start`, `coupon_manual_input`, `coupon_image_upload`, `coupon_auto_scan`, `coupon_auto_success`, `coupon_auto_fail`, `coupon_register_complete` |
| 쿠폰 액션 | `coupon_detail_view`, `coupon_use_complete`, `coupon_edit`, `coupon_delete`, `coupon_move_to_membership`, `coupon_expired` |
| 멤버십 | `membership_add`, `membership_list_view`, `membership_detail_open`, `membership_detail_close`, `membership_edit`, `membership_delete` |
| 알림 | `push_received`, `push_open`, `push_click` |
| 다이얼로그 | `dialog_register_confirm/cancel`, `dialog_use_confirm/cancel`, `dialog_edit_confirm`, `dialog_delete_confirm/cancel` |
| 홈/리포트 | `home_view`, `report_banner_view`, `report_banner_click` |
| 기타 | `coupon_filter`, `coupon_search`, `coupon_share`, `notification_list_view`, `settings_view`, `settings_notification_toggle`, `settings_notification_timing_toggle` |

### Q. `memberships`의 브랜드는 `brands` 테이블의 브랜드와 다른 건가요?
`brands.csv`의 쿠폰 브랜드는 14개이고, `memberships.csv`의 멤버십 브랜드는 10개입니다. 일부 겹치는 브랜드(올리브영, CU, GS25, 스타벅스)도 있고, 멤버십에만 있는 브랜드(이마트, 네이버페이 등)도 있습니다.

## 데이터 품질

### Q. `event_logs.session_id`가 비어있는 이벤트가 있어요. 버그인가요?
의도적으로 넣은 노이즈입니다. 실제 서비스에서도 세션 추적 실패가 발생하며, 전처리 시 어떻게 처리할지 판단이 필요합니다.

### Q. `event_logs`에서 같은 `event_id`가 2번 나와요.
이것도 의도된 노이즈입니다. 실무에서 네트워크 재시도 등으로 중복 로깅이 발생하는 상황을 재현한 것이며, 분석 전 dedup 처리가 필요합니다.

### Q. 그 외에 알아야 할 노이즈가 있나요?
일부 이벤트에 타임존 오프셋으로 인한 미세한 시간 편차가 있습니다. 분석 전 EDA를 먼저 수행하여 데이터 상태를 파악하는 것을 권장합니다.

## 분석 시작

### Q. 퍼널 분석은 어떤 순서로 보면 되나요?

**신규 유저 여정:**
`app_install` → `app_signup_start` → `app_signup_complete` → `app_onboarding_complete` → `coupon_register_start` → `coupon_register_complete` → `coupon_use_complete`

**푸시 알림 퍼널:**
`push_received` → `push_open` → `push_click` → `coupon_detail_view` → `coupon_use_complete`

### Q. 코호트 분석은 어떻게 시작하나요?
`users.signup_at`에서 가입 주(week)를 추출하고, `event_logs`에서 주별 활동 여부를 매핑하면 됩니다.

### Q. A/B 테스트는 어떻게 분석하나요?
`ab_assignments`에 유저별 실험 배정 정보(`experiment_name`, `variant`)가 있습니다. `event_logs`와 `user_id`로 JOIN하여 control/treatment 그룹 간 지표를 비교하면 됩니다.
