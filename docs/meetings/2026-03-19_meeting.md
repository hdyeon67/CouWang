## 회의 목적
- 프로젝트 방향성을 빠르게 정리하고, 앱 구현과 데이터 분석의 우선순위를 정한다.
- 쿠폰 관리 서비스의 기본 가치와 확장 가능성을 검토한다.

## 주요 논의 내용

### 1. 프로젝트 기본 방향
- 쿠폰 만료 방지를 돕는 서비스로 출발한다.
- 서버 연결 없이 로컬 저장과 CSV 기반 분석 흐름으로 진행한다.
- Flutter 앱, SQLite, Python 분석, 대시보드까지 연결하는 포트폴리오형 구조를 목표로 한다.

### 2. 데이터 분석과 구현의 균형
- 팀 전체 결과물 관점에서 데이터 분석 비중이 중요하다는 의견이 나왔다.
- 앱 구현 자체보다도 이벤트 수집, 퍼널 설계, 대시보드와 분석 결과 재현이 핵심 자산이 될 수 있다고 봤다.
- 다만 완전한 분석 프로젝트로만 가기보다, 설명 가능한 MVP 화면도 필요하다는 점을 함께 확인했다.

### 3. 서비스 가치와 확장성 검토
- 단순히 쿠폰을 보관하는 앱보다, 만료 전 사용을 유도하는 서비스여야 한다는 점이 반복해서 언급됐다.
- 쿠폰 이미지 자동 입력, OCR, 스캔 기능은 매력적이지만 구현 난이도가 높아 MVP 범위를 신중하게 봐야 한다고 정리했다.
- 오프라인 쿠폰까지 포함할 수 있는 구조가 더 현실적이라는 의견이 나왔다.

### 4. 고객/타겟 정의 필요성
- 기능보다 먼저 타겟 고객, 페르소나, 고객이 느끼는 가치가 정리되어야 한다는 피드백이 나왔다.
- 경쟁 서비스와 비교했을 때 왜 이 서비스여야 하는지 설명 가능한 차별점이 필요하다는 점을 확인했다.

### 5. 브랜드와 네이밍
- 서비스 이름 후보를 여러 개 논의했고, 기억하기 쉽고 캐릭터화하기 쉬운 이름이 필요하다는 공감대가 있었다.
- 강아지 캐릭터를 활용한 친근한 브랜드 방향이 자연스럽게 연결될 수 있다는 의견이 나왔다.

### 6. 화면 및 기능 초기 아이디어
- 홈 화면은 리스트 중심으로 단순하게 가는 쪽이 낫다는 의견이 나왔다.
- 설정 화면은 꼭 별도 탭일 필요가 없고, 알림과 직접 연결되는 편이 자연스럽다는 논의가 있었다.
- 푸시 알림은 서비스 존재 이유와 직접 연결되기 때문에 기능 구조상 중요하게 다뤄야 한다고 봤다.

## 핵심 결정 사항
- 프로젝트는 쿠폰 만료 방지 서비스로 시작한다.
- 로컬 저장, 샘플 데이터, 이벤트 로그, SQL/대시보드 분석 흐름을 함께 가져간다.
- 고객 가치와 핵심 차별점을 먼저 정리한 뒤, 화면과 이벤트 구조를 구체화한다.
- 데이터 분석 관점에서 불필요한 기능은 과감히 제외한다.

## 남은 쟁점
- 앱 구현과 데이터 분석 중 어디에 더 무게를 둘지 최종 비중 조정 필요
- OCR/자동 입력을 MVP 범위에 포함할지 여부
- 타겟 고객과 페르소나를 어느 수준까지 구체화할지 결정 필요
