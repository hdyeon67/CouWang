- iOS Firebase 설정 파일 `GoogleService-Info.plist`는 저장소 배치만으로 끝내지 않고, Runner 타깃 Resources에 포함되어 실제 `Runner.app` 번들에 복사되어야 한다.

- 단, QA나 Crashlytics 확인이 필요할 때는 `ENABLE_INTERNAL_TEST_TOOLS=true` Dart define으로만 임시 노출한다.

### iOS 업데이트 빌드

## 2026-04-21

### 법적 문서와 공개 URL
- 스토어 제출에 필요한 개인정보처리방침은 `docs/legal/`에 원문 Markdown과 GitHub Pages용 HTML을 함께 관리한다.
- GitHub Pages는 저장소의 `docs/` 폴더 기준 배포를 전제로 하며, 대표 개인정보처리방침 URL은 `https://hdyeon67.github.io/CouWang/legal/privacy_policy_ko.html`로 둔다.
- 쿠왕 v1.0은 로그인과 쿠왕 자체 서버 저장이 없고 쿠폰/멤버십 데이터는 기기 내 로컬 저장을 기준으로 한다.
- Google AdMob 배너 광고를 적용했으므로 앱 개인정보 라벨은 `데이터 수집함` 기준으로 갱신한다. 광고 식별자, 사용 데이터, 진단 정보 등 Google Mobile Ads SDK가 처리할 수 있는 항목은 스토어 제출 전 최신 Google 공시 문서 기준으로 확인한다.
- 사진/이미지는 현재 기기 내 로컬 저장만 하므로 App Store 개인정보 라벨에서는 수집 데이터로 보지 않는다. 향후 외부 서버 전송이 생기면 사용자 콘텐츠 항목을 재검토한다.

### 화면 방향
- 쿠왕 앱은 모바일 쿠폰/멤버십 관리 경험을 세로 화면 기준으로 고정한다.
- Flutter, Android, iOS 설정을 모두 세로 고정으로 맞춰 플랫폼별 회전 차이를 줄인다.

### 광고와 Firebase
- Google AdMob은 실제 Android/iOS App ID와 배너 광고 단위 ID를 코드에 반영한다.
- 광고 SDK가 포함되므로 스토어 개인정보 문서는 `데이터 수집함` 기준으로 관리한다.
- Firebase Analytics/Crashlytics SDK는 포함하되, 실제 수집은 Firebase 설정 파일과 `ENABLE_FIREBASE=true` 빌드 옵션이 있을 때만 활성화한다.
- Firebase 설정 파일이 없는 기본 릴리즈 빌드는 Firebase 수집 비활성 상태로 유지한다.

### 내부테스트 AAB
- 내부테스트용 Android App Bundle은 `flutter build appbundle --release`로 생성한다.
- 2026-04-21 생성본은 `1.1.5+8`, SHA-256 `a9352f688e33c88429b49137c02b78fbc965cd26a66d7d2ee3abb9f5134f8bea` 기준으로 기록한다.

### 문서 구조
- `docs/`는 기획, 아키텍처, 분석, 스토어, 법무, 운영, 작업 노트, 회의록 폴더로 나누어 관리한다.
- 이벤트 명세와 KPI 정의는 `docs/analytics/`에 둔다.
- 실행 및 릴리즈 가이드는 `docs/operations/`에 둔다.
- GitHub Pages 공개 URL에 쓰는 HTML 문서는 기존 경로를 유지한다.

## 2026-04-20

### 스토어 제출 준비
- 플레이스토어/앱스토어 등록정보와 이미지 기획서는 `docs/store/`에 별도 산출물로 관리한다.
- 스토어 문구는 현재 구현된 기능인 로컬 저장, 로그인 없음, 갤러리 이미지 기반 OCR/바코드/QR 인식, 쿠폰/멤버십 관리, 로컬 알림을 기준으로 작성한다.
- 개발사명, 연락처, 개인정보처리방침 URL처럼 실제 제출 전 확정이 필요한 값은 문서에 `(기입 필요)`로 남긴다.

### 쿠폰 유효기간 입력 정책
- 사용자는 지난 유효기간도 직접 선택할 수 있다.
- OCR/이미지 인식으로 과거 날짜가 추출된 경우에도 등록 폼에 반영한다.
- 과거 유효기간 쿠폰은 저장 후 만료 상태로 분류하며, 만료된 쿠폰은 신규 알림 예약 대상에서 제외한다.

### iOS 알림 클릭 라우팅
- iOS에서는 알림 payload가 앱 초기화 또는 스플래시 완료 전에 들어올 수 있으므로, 네비게이터가 준비되지 않은 경우 payload를 pending 상태로 보관한다.
- pending payload는 첫 프레임 이후 또는 앱 resume 시 다시 처리해 해당 쿠폰 상세 화면으로 이동한다.
- 알림 payload 형식은 기존 결정대로 `couponId|type`을 유지하고, 과거 호환용 `couponId` 단독 payload도 계속 허용한다.

### 스토어 캡처용 테스트 데이터
- 설정 화면에 개발/QA용 테스트 쿠폰 추가와 가상 멤버십 추가 버튼을 둔다.
- 테스트 쿠폰의 화면 표시명에서는 `내부테스트` 문구를 제거해 스토어 스크린샷에 자연스럽게 보이도록 한다.
- 가상 멤버십은 OK캐시백, 해피포인트, L.POINT, CJ ONE 4종을 고정 ID로 저장해 중복 생성하지 않는다.
- 설정 화면의 알림 섹션과 테스트 버튼 섹션 사이 여백은 스토어 캡처를 위해 크게 벌릴 수 있으며, 출시 전 운영 UI 유지 여부를 다시 확인한다.

## 2026-04-14

### Android 알림 권한 정책
- Android 13 이상은 시스템 알림 런타임 권한을 사용한다.
- Android 12 이하는 시스템 알림 권한 팝업이 없으므로, 앱 내부 안내 다이얼로그로 사용자 동의를 받고 내부 알림 설정을 켠다.
- 첫 설치 기본값은 알림 `OFF`로 두고, 사용자가 명시적으로 동의한 뒤에만 알림 설정을 `ON`으로 저장한다.
- 시스템 권한 허용 여부와 앱 내부 알림 설정 ON/OFF는 서로 다른 개념으로 보고 설정 화면에서도 분리해 다룬다.

### 웹 실행 지원
- Chrome 실행은 모바일 핵심 기능 검증용이 아니라 구조 확인용으로 유지한다.
- 웹에서 지원되지 않는 로컬 알림 플러그인 호출은 앱 초기화와 화면 진입을 막지 않도록 건너뛴다.
- SQLite wasm은 패키지 버전과 맞는 바이너리를 프로젝트 자산으로 명시 관리한다.

### 공통 UI 보정
- 리스트 empty 상태는 빈 화면이나 제각각의 원형 자리 대신 공통 마스코트 empty UI로 맞춘다.
- 검색 필드는 바깥 컨테이너의 라운드 border가 깨져 보이지 않는 것을 우선해 내부 TextField 크기를 살짝 줄여 맞춘다.

## 2026-04-13

### 알림 인박스 / 테스트 알림
- 알림 재스케줄은 사용자가 보는 알림 로그를 삭제하지 않는 방향으로 분리한다.
- 알림 클릭 payload는 쿠폰 ID만으로는 여러 알림 중 어떤 항목을 눌렀는지 구분하기 어려우므로 `couponId|type` 형식으로 확장하되, 기존 payload와의 호환은 유지한다.
- 테스트 알림은 실제 OS 예약 알림을 우선 유지하되, 앱이 foreground 상태일 때도 확인할 수 있도록 설정 화면에서 보조 즉시 알림 흐름을 함께 둔다.

### 권한 / 심사 전략
- 쿠왕 MVP는 라이브 카메라 스캔을 제공하지 않고 갤러리 이미지 기반 OCR/바코드/QR 인식만 제공하므로, iOS 카메라/Bonjour/로컬 네트워크 권한 설명은 제거한다.
- 첫 심사 전에는 AdMob을 붙이지 않고, 광고 SDK와 개인정보/광고 선언으로 인한 심사 변수를 줄인 뒤 추후 버전에서 붙이는 방향을 우선한다.
- 스토어 문구에는 `실시간 카메라 스캔`처럼 현재 앱 기능과 다른 표현을 쓰지 않는다.

### 배포 관리
- Android 내부 테스트용 AAB는 release signing 설정과 버전 코드 증가를 전제로 관리한다.
- 실제 keystore 비밀번호 파일은 저장소에 넣지 않고, 예시 파일과 README 안내만 커밋한다.
- 앱 내 설정 화면에는 사용자가 크게 의식하지 않아도 되는 위치에 현재 버전명/빌드 번호를 표시한다.

## 2026-04-07

### iOS 배포 준비
- TestFlight 배포를 염두에 두고 iOS archive 기준의 빌드/서명/아이콘 요구사항을 우선 정리한다.

- iOS 배포 검증은 `flutter build ios --no-codesign` 통과 후 TestFlight 내부 설치까지 확인하는 흐름으로 가져간다.

## 2026-04-06

### 데이터 저장 방식
- 현재 MVP는 화면용 가상 데이터가 아니라 실제 로컬 데이터 기반으로 동작해야 하므로, 쿠폰/멤버십/알림 설정은 로컬 SQLite에 저장한다.
- 이미지 파일은 대용량 바이너리를 그대로 DB에 넣기보다 메타데이터와 파일 경로를 분리해 관리한다.
- 향후 로그인/클라우드 백업 확장을 고려하되, 현재 구조는 로컬 단독 사용 기준으로 단순하고 안정적으로 유지한다.

### 알림
- 알림 리스트는 쿠폰 상태를 바탕으로 계산한 가상 목록이 아니라, 실제 알림 로그를 저장하는 인박스형 구조로 관리한다.
- 읽은 알림과 삭제된 알림 상태는 별도 로그 데이터로 관리한다.
- 쿠폰 사용 완료, 쿠폰 삭제, 알림 설정 변경은 로컬 알림 스케줄 재정리와 연결한다.

### 웹 대응
- Chrome 환경에서도 구조를 확인할 수 있도록 웹 실행 지원을 유지한다.
- 웹 DB는 SQLite wasm 자산을 프로젝트에서 명시적으로 관리하는 방향으로 정리한다.

## 2026-03-23

### 제품 방향
- 이번 MVP는 복잡한 AI 기능보다 실제로 동작하는 쿠폰/멤버십 관리 흐름에 집중한다.
- 홈 화면과 쿠폰 리스트 화면은 하나의 화면으로 통합한다.
- 사용자는 앱에 들어오자마자 현재 보유 쿠폰과 만료 임박 상태를 바로 확인할 수 있어야 한다.

### 홈 화면
- 홈은 `쿠폰 사용 유도형 대시보드` 성격으로 정리한다.
- 상단에는 마스코트와 절약 리포트 메시지를 배치한다.
- 쿠폰 리스트 기본 정렬은 만료 임박순으로 둔다.
- 임박 쿠폰은 D-day 배지와 카드 강조로 눈에 띄게 표현한다.
- 브랜드 키 컬러는 `#64CAFA`를 기준으로 사용한다.

### 쿠폰 등록
- 수동 입력을 기본 등록 방식으로 유지한다.
- 이미지 업로드 기반 자동 입력은 보조 흐름으로 제공한다.
- 바코드/QR 코드 인식과 OCR은 등록 보조 기능으로 연결한다.
- 등록 필수값은 너무 무겁지 않게 유지하되, 이미지 첨부의 중요성은 높게 본다.

### 쿠폰 상세
- 상세 화면에서는 D-day와 유효기간을 명확히 보여준다.
- 이미지 확대와 바코드 확인이 가능해야 한다.
- 수정/삭제는 메인 CTA보다 보조 관리 액션으로 둔다.

### 멤버십/알림
- 멤버십은 별도 리스트 구조로 유지한다.
- 알림은 단순 ON/OFF가 아니라 세부 주기 조정이 가능한 구조를 지향한다.
- 기능 위치는 설정 탭 존재 자체보다 사용자가 쉽게 찾을 수 있는 흐름을 우선한다.

### 구현 기준
- 버튼, 필드, 메뉴 명칭은 문서와 화면에서 통일한다.
- 기능을 늘리기보다 기존 화면을 더 직관적으로 다듬는 것을 우선한다.
- 분석 데이터는 실제 로그가 아닌 합성 데이터라는 점을 문서와 발표에서 명확히 구분한다.
